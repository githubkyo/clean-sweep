# Macの「システムデータ 274GB」の正体を暴いたら、半分以上ゴミだった話

Macのストレージがパンパンで、新しいアプリも入れられない。設定画面を開くと「システムデータ」が**274GB**。460GBのSSDの6割をシステムデータが占めている。

「いや、OSってそんなに重くないだろ......」

そう思って調べ始めたら、想像以上にカオスな世界が広がっていました。

---

## きっかけ：Xcodeのアップデートすらできない

ある日、Xcodeのアップデート通知が来たので実行しようとしたら「ストレージが不足しています」と言われました。Mac Mini（M2、460GB SSD）で開発しているのですが、残りがほぼゼロ。

ストレージの内訳を見ると、「システムデータ」が274GB。写真や書類はたいしてないのに、なぜこんなことに？

とりあえずClaude Codeを使ってターミナルからディスクの中身を調査してみることにしました。

## 犯人その1：Docker（104GB）

最初に見つかった巨大なゴミの山が**Docker**でした。

- 未使用のDockerイメージ：**69GB**
- ボリューム：**14GB**
- ビルドキャッシュ：**21GB**

開発でいろんなコンテナを試しては放置、試しては放置を繰り返した結果です。`docker system df`で確認すると、使っていないイメージだけで69GB。

```
docker system prune -a --volumes
```

このコマンド一発で**48GB**回収。気持ちいい。

## 犯人その2：Xcode関連ファイル（58GB）

iOS開発者なら心当たりがあるはず。Xcodeは黙っていると大量のキャッシュやログを溜め込みます。

| フォルダ | サイズ |
|---|---|
| DerivedData（ビルドキャッシュ） | 5.7GB |
| iOS DeviceSupport | 16GB |
| Archives（過去のビルド） | 4.7GB |
| CoreSimulator | 6.8GB |

iOS DeviceSupportは、実機をつなぐたびにデバイスのバージョンごとにシンボル情報をダウンロードするもの。古いiOSバージョンのものはまず使いません。

DerivedDataは消しても次のビルドで再生成されるので、気軽に削除できます。Archivesも過去にApp Storeに提出済みのものなら不要。

合計で**14.3GB**回収しました。

## 犯人その3：Claude Codeの一時ファイル（84GB）

ここが一番衝撃でした。

`/private/tmp/claude-501`というディレクトリに、なんと**84GB**ものファイルが残っていました。

正体は、Claude Codeがgit worktreeを使って作業した際の残骸。あるプロジェクトのworktreeが丸ごとコピーされたまま放置されていたのです。

AIコーディングツールは便利ですが、裏側でこういったファイルを生成していることがあります。`/private/tmp`は再起動で消えるはずですが、Macをスリープ運用していると溜まり続けます。

**84GB**、まるごと回収。

## 犯人その4：npm / Gradle / 各種パッケージキャッシュ

開発者のMacには、いろんなパッケージマネージャのキャッシュが積もっています。

- npm：**11GB**
- Gradle：**8.1GB**
- Yarn：**2.5GB**

これらは `npm cache clean --force` や Gradleのキャッシュディレクトリ削除で対応できます。ただし、次のビルド時にダウンロードし直しになるので、回線が細い環境では注意。今回は**約5GB**だけ選択的に削除しました。

## その他：消せないもの・消しにくいもの

調査で見つかったけれど簡単には消せないものもありました。

- **APFSスナップショット（数十GB）**：OSアップデート時に自動作成される。sudo権限が必要で、下手に消すとシステムに影響が出る可能性あり
- **Preboot（37GB）**：OSアップデートの準備領域。アップデート完了後に自動解放されるので放置が正解
- **VM（26GB）**：仮想メモリ。macOSが管理しているので削除不可

## 最終結果：156GB回収

| 対象 | 回収量 |
|---|---|
| Docker | 48GB |
| Xcode関連 | 14.3GB |
| Claude Code一時ファイル | 84GB |
| 各種キャッシュ | 約5GB |
| その他 | 約5GB |
| **合計** | **約156GB** |

460GBのSSDで156GB回収。ストレージの3分の1が戻ってきた計算です。

## この経験で気づいたこと

ストレージ圧迫の原因は、ほとんどが「知らないうちに溜まったキャッシュや一時ファイル」でした。macOSの設定画面では「システムデータ」としか表示されず、中身がわからない。だから放置してしまう。

でも実際にターミナルで調べてみると、安全に消せるものが大半です。

問題は、**毎回ターミナルでコマンドを打つのは面倒**ということ。どのディレクトリが消せてどれが危険なのか、毎回調べるのも手間です。

## CleanSweep：開発者のための自動お掃除ツール

そこで、この経験をもとに**CleanSweep**というmacOSアプリを作りました。

開発者のMacにありがちなゴミを自動検出して、安全に削除できるツールです。SwiftUIで作っていて、GitHubでオープンソース公開しています。

**主な機能：**
- Docker、Xcode、npm、Gradleなどの開発キャッシュを一括スキャン
- 各項目のサイズと安全性レベルを表示
- ワンクリックで選択削除
- 削除前の確認と、何が消えるかの説明付き

GitHub: [https://github.com/githubkyo/clean-sweep](https://github.com/githubkyo/clean-sweep)

ターミナルが苦手な人でも、GUIで安全にストレージを回収できます。「Macが重い」「ストレージがいっぱい」と感じている開発者の方は、ぜひ試してみてください。
